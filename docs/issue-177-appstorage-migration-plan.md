# Issue #177: `SwiftyUserDefaults` dependency survey and `@AppStorage` migration plan

## Scope

This note documents the current `AppSettingModel` persistence surface, all direct read/write locations found in the repo, `@AppStorage` applicability by type, and a phased migration plan for follow-up issues.

Out of scope:

- actual migration implementation
- removal of `SwiftyUserDefaults`
- UI changes

## Current topology

- `AppSettingModel` itself only declares `isActiveByCommand` directly in `ZIGSIMPlus/Models/AppSettingModel.swift`.
- 27 persisted properties are generated into `ZIGSIMPlus/Autogenerated/AppSettingModelAutoSave.generated.swift`, not `ZIGSIMPlus/Models/AutoSave.generated.swift` as written in the issue.
- The generated extension uses two mechanisms per property:
  - in-memory caching via associated objects
  - persistence via `Defaults[key: DefaultsKeys.*]`
- Command activation state is a separate path:
  - in-memory dictionary: `AppSettingModel.isActiveByCommand`
  - persistent keys: `DefaultsKeys.is*CommandActive`
  - key mapping: `Command.userDefaultsKey`

## Migration implications

### What is easy

- 27 generated properties can be mapped one-for-one to `@AppStorage` or an equivalent wrapper while preserving existing keys and defaults.
- All custom setting enums already have `Int` raw values, so no serialization redesign is needed.

### What is not easy

- `AppSettingModel` is a UIKit-era singleton, not a SwiftUI `View` or `ObservableObject`.
- The current generated layer provides both cached access and persistence in one place. A direct in-place annotation swap would change ownership and observation semantics at the same time.
- `isActiveByCommand` is not a single persisted value. It is an adapter over 18 individual boolean keys and is read from many services, so it needs an explicit compatibility layer.

## Recommended migration shape

### Phase 1

Introduce a storage adapter that preserves current call sites and keys.

- Keep `AppSettingModel.shared` as the public facade initially.
- Move persistence behind a small storage layer instead of annotating the singleton directly in the first step.
- Preserve every existing `UserDefaults` key to avoid data loss.

### Phase 2

Migrate the low-risk core settings first.

- `dataDestination`
- `transportProtocol`
- `ipAddress`
- `portNumber`
- `transportFormat`
- `messageRatePerSecond`
- `deviceUUID`

Reason: these have clear read/write paths and limited UI surface in `CommandSettingPresenter`.

### Phase 3

Migrate command detail settings by feature area.

- compass and beacon
- NDI
- ARKit
- image detection

Reason: each area is already clustered in `CommandDetailSettingsPresenter`.

### Phase 4

Replace command activation persistence.

- Introduce explicit per-command accessors over the 18 boolean keys.
- Keep `isActiveByCommand` as a computed compatibility facade until service consumers are updated.
- Remove `Command.userDefaultsKey` only after all readers stop using `DefaultsKey<Bool>`.

### Phase 5

Delete legacy generation and dependency.

- remove Sourcery template/generation for `AppSettingModelAutoSave`
- remove remaining `SwiftyUserDefaults` imports and package dependency
- run device validation on settings-edit flows after the storage swap

## Proposed follow-up issue split

1. Add a new settings storage facade and migrate the 7 core settings without changing public behavior.
2. Migrate NDI, compass, beacon, ARKit, and image-detection detail settings in feature-grouped slices.
3. Replace command activation persistence with per-command storage accessors plus a compatibility dictionary facade.
4. Remove Sourcery-generated autosave code and `SwiftyUserDefaults`, then run final regression and real-device validation.

## Risks

- `@AppStorage` ownership: direct use inside the current plain class risks mixing storage migration with lifecycle and observation changes.
- Backward compatibility: the keys `messageFormat`, `ndiType`, `userImageDetectorCameraPosition`, and `userImageDetectorResolution` must stay unchanged.
- Command activation fan-out: `isActiveByCommand` is read broadly across service classes, so removing it outright would create a wide regression surface.
- Device behavior: even though this issue is documentation-only, the eventual migration will need real-device validation for sensor-driven commands.

- No source migration was performed.
- No dependency removal was performed.
- No runtime behavior was changed.
